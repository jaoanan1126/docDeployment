A **bill** is the structured accounts payable record in Ramp—the object that powers approvals, coding, and payment.

For product details, see the [Ramp Help Center](https://support.ramp.com/hc/en-us/articles/4417743160211-Bill-Pay-overview).

### Common use cases

- **ERP integration**: Sync bills and payments between Ramp and your accounting system
- **Bill processing automation**: Programmatically create bills that are automatically approved for payment
- **Financial reporting**: Extract bill and payment data for custom reporting and analytics

---

## How Bills Work

### Bills and payments

Payments are nested within bills and do not have their own dedicated endpoints. When you create a bill via API, you include payment details in the same request. Payment info is returned as part of the bill object on `GET /bills`.

Payments can be created alongside bills, but have limited support for updates once created.

### Bill object lifecycle

Bills move through the following statuses:

1. **Draft** → Invoice uploaded, OCR extracts details
2. **Pending approval** → Awaiting approver sign-off
3. **Approved** → Ready for payment. Bills follow your customer's configured [approval policy](https://support.ramp.com/hc/en-us/articles/4417843897747-Bill-Pay-approvals) in Ramp.
4. **Paid** → Payment processed

Bills created via API are automatically approved and enter the workflow at the **Approved** status.

<Info>
**Draft bill endpoints**

We have draft bill endpoints available to create and update draft bills through the API. Draft bills must be approved through the Ramp dashboard—approval via API is not supported. [Submit a Developer API support ticket](/developer-api/v1/debugging#how-to-submit-a-developer-api-support-ticket) for early access.
</Info>

## Create Bill Request Builder

Use the [POST /bills](/api-reference/bill/create-a-bill#post-developer-v1-bills) endpoint to create a bill. This covers the entire bill object, including the bill payment.

The request requires several UUIDs. Fetch these from the endpoints below and use the response field value as the value for the request field.

| Source endpoint | Response field | Request field (for POST /bills) |
| --- | --- | --- |
| [GET /entities](/api-reference/business-entities/list-business-entities#get-developer-v1-entities) | `id` | `entity_id` |
| [GET /entities](/api-reference/business-entities/list-business-entities#get-developer-v1-entities) | `bank_accounts[].id` where `usage_type = BILL_PAY_BANK_ACCOUNT` | `source_bank_account_id` |
| [GET /vendors](/api-reference/vendor/list-vendors#get-developer-v1-vendors) | `id` | `vendor_id` |
| [GET /vendors/&#123;id&#125;/contacts](/api-reference/vendor/list-vendor-contacts-for-vendor#get-developer-v1-vendors-vendor-id-contacts) | `id` | `vendor_contact_id` |
| [GET /vendors/&#123;id&#125;/accounts](/api-reference/vendor/list-vendor-bank-accounts#get-developer-v1-vendors-vendor-id-accounts) | `id` (for ACH/wire/SWIFT only) | `vendor_account_id` |

<Info>
**Missing BILL_PAY_BANK_ACCOUNT?**

If you don't see a bank account with `usage_type = BILL_PAY_BANK_ACCOUNT` in your GET /entities response, Bill Pay hasn't been set up for your organization yet. Follow the steps in the [Bill Pay set up guide](https://support.ramp.com/hc/en-us/articles/4417760908435-Bill-Pay-set-up) to configure it, then retry your API call.
</Info>

## Payment Methods

### Card

Paying by Ramp Card earns cashback on bill payments. Two options:

**Existing Ramp Card** – Use already-created cards for bill payments. Manually share card details with vendors, then use transaction matching to link the payment to the bill once it clears.

**Single-Use Virtual Card** – Ramp generates a unique virtual card for each bill with the limit set to the bill amount. Two delivery options:

- **Card delivery disabled** – Card details become available once the bill is approved; manually share with vendors
- **Card delivery enabled** – Ramp automatically emails card details to the vendor's primary contact on the due date, with reminders at 2, 4, and 7 days

Single-use cards auto-lock when the spending limit is reached, after 14 days (delivery enabled) or 21 days (delivery disabled). Transaction matching happens automatically once payments clear. Supports international bills in local currency.

For programmatic access to full card details (PAN/CVV) or custom spending controls, see the [Virtual Cards guide](/developer-api/v1/virtual-cards).

To filter out one-time card transactions in the Transactions API, set `has_no_sync_commit=true`.

### ACH, Check, Wire, and SWIFT

These payment methods require a `source_bank_account_id` from [GET /entities](/api-reference/business-entities/list-business-entities#get-developer-v1-entities).

| Method | Requirements | Timing |
|--------|--------------|--------|
| **ACH** | Verified vendor bank account | 2–3 business days |
| **Check** | Vendor address | 5–7 business days |
| **Wire** | Verified vendor bank account | Same-day (domestic) |
| **SWIFT** | Verified vendor bank account | Varies (international) |

- Payments arrive by `payment_arrival_date` (cannot be a weekend or bank holiday)
- Non-check payments require a verified vendor bank account via [GET /vendors/&#123;id&#125;/accounts](/api-reference/vendor/list-vendor-bank-accounts#get-developer-v1-vendors-vendor-id-accounts). See [Create Bill Request Builder](#create-bill-request-builder) for full schema details.
- Check payments require a vendor address on file
- Expedited ACH and Wire payments are not available via API

## Syncing to Your ERP

Bills sync automatically with direct accounting connections (QuickBooks, NetSuite, etc.). For custom integrations, see the [Accounting guide](/developer-api/v1/accounting#example-bill-and-bill-payment-sync).

Query with `sync_ready=true` to find bills ready for export, then use `sync_status` and the returned bill `status` together to determine whether to sync the bill, the payment, or both.

<Warning>
**UCSV restriction**

Bills cannot be synced via API with Universal CSV. Use the [dashboard export](https://support.ramp.com/hc/en-us/articles/31817848602131-Exporting-Bills-via-CSV) instead.
</Warning>

## 3-Way Match

To verify a bill matches its purchase order and item receipt:

1. **Get the bill** → extract `purchase_order_ids`, `line_items[].purchase_order_line_item_id`, and `inventory_line_items[].purchase_order_line_item_id`
2. **Get the purchase order** → extract `item_receipt_ids`
3. **Get the item receipt** → confirm `line_items[].purchase_order_line_item_id` matches the bill

When all three documents share the same line item IDs, you have a verified match.

## Current API Limitations

### Approval process management

- You cannot trigger or manage bill approvals through the API
- You cannot mark a draft bill as approved or configure approval process logic via API
- Bills created via API are automatically approved

### Payment management

- Payments are nested within bills and do not have their own dedicated endpoints
- Payments can be created alongside bills, but have limited support for updates once created
- Batch payments (creating a single payment for multiple bills) are not supported

### Document and file management

### Document and file management

- OCR upload for draft bills is not currently supported via API
- Images of invoices cannot be attached through the API

### Other limitations

- Managing entity bank accounts is not supported via API (vendor bank accounts can be managed through [vendor accounts endpoints](/api-reference/vendor/list-vendor-bank-accounts#get-developer-v1-vendors-vendor-id-accounts))

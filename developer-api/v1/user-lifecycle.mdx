## Overview

This guide covers three example workflows for managing Ramp users over the Developer API:

1. [Onboard, update, and deprovision a user](#scenario-1-onboard-update-and-deprovision-a-user)
2. [Deactivate and reactivate a user](#scenario-2-deactivate-and-reactivate-a-user)
3. [Stage a draft user and schedule their invite for later](#scenario-3-stage-a-draft-user-and-schedule-their-invite)

Full endpoint schemas live in the [Users API reference](https://docs.ramp.com/developer-api/v1/api/users). The [Reference](#reference) section covers the status model and listing/filtering patterns.

<Info>
**Works on users from any source**

Users created outside the Developer API — via HRIS or SCIM sync, a manual invite from the Ramp app, etc. — can still be deactivated, scheduled for deactivation, and reactivated through these endpoints. `GET /developer/v1/users/{user_id}` (or the list endpoint) to see their current state, then apply the relevant scenario below.
</Info>

### Before you start

- You need an OAuth access token with the `users:read` and `users:write` scopes. See the [Authorization guide](https://docs.ramp.com/developer-api/v1/authorization) for how to obtain one.
- If you plan to set a department, location, or manager on new users, look those up first via [GET /departments](https://docs.ramp.com/developer-api/v1/api/departments#get-developer-v1-departments), [GET /locations](https://docs.ramp.com/developer-api/v1/api/locations#get-developer-v1-locations), and [GET /users](https://docs.ramp.com/developer-api/v1/api/users#get-developer-v1-users).

### How asynchronous user creation works

User creation is asynchronous. `POST /developer/v1/users/deferred` returns a task id; poll `GET /developer/v1/users/deferred/status/{task_id}` every 1–2 seconds until `status` is `SUCCESS` (possible values: `STARTED`, `IN_PROGRESS`, `ERROR`, `SUCCESS`). Most tasks finish in under 5 seconds.

A terminal status response looks like:

```json
{
  "id": "b39ab676-3faa-4447-ba66-906860219fdf",
  "status": "SUCCESS",
  "data": {"user_id": "019db12c-a1e9-7f6e-afc8-2cb64d7f86f7"},
  "context": {"acting_user_id": "ea8213c3-6b99-49c5-a45f-0b48d66ae0ea"}
}
```

On `SUCCESS`, `data.user_id` is the UUID of the created user.

## Scenario 1: Onboard, update, and deprovision a user

Invite a new hire, keep their profile in sync as the org changes around them, and deactivate them when they leave. The three endpoints here are `POST /users/deferred`, `PATCH /users/{user_id}`, and `PATCH /users/{user_id}/deactivate`.

### Invite them on day one

For an employee starting today, call `POST /developer/v1/users/deferred` with `role` and their profile fields. This sends the invite email immediately. A typical request body:

```json
{
  "email": "jordan.manager@company.com",
  "first_name": "Jordan",
  "last_name": "Manager",
  "role": "BUSINESS_USER",
  "department_id": "29693a31-0f8d-4c91-be40-016bc0a6161c",
  "location_id": "1552c5a2-8889-4c91-b557-25419e0c616e",
  "direct_manager_id": "517347b0-0841-4e3f-9b78-45cdb9642f5c",
  "is_manager": true,
  "idempotency_key": "hris-sync-2026-04-21-jordan-manager"
}
```

The response is the deferred-task handle: `{ "id": "165c7dc4-7b6b-4ee5-baa6-1f28701b075e" }`.

Only `email`, `first_name`, `last_name`, `role`, and `idempotency_key` are required — if you omit department and location, Ramp assigns the business's defaults. The fields you can send on create:

| Field | Notes |
| --- | --- |
| `role` | One of `BUSINESS_USER`, `BUSINESS_ADMIN`, `BUSINESS_BOOKKEEPER`, `IT_ADMIN`, `AUDITOR`, `GUEST_USER`. `BUSINESS_OWNER` cannot be invited via the API. |
| `department_id`, `location_id` | Ramp UUIDs. `location_id` also determines the user's business entity (`entity_id` on the response). |
| `direct_manager_id` | Ramp UUID of an existing user. The request uses `direct_manager_id`; the response calls it `manager_id`. |
| `is_manager` | Whether this user can approve spend and manage reports. |
| `scheduled_deactivation_date` | In `YYYY-MM-DD` format. Useful if you already know the user's end date — see [Schedule their last day](#schedule-their-last-day). Cannot be set for admins or owners. |
| `idempotency_key` | A client-generated unique string. Ramp rejects duplicate keys with `400 DEVELOPER_7005`; generate a new key for every create attempt. |

Poll the deferred task until it reports `SUCCESS`, then `GET /developer/v1/users/{user_id}` to fetch the user:

```json
{
  "id": "019db12c-faa3-772e-b874-7e8bce265bbb",
  "email": "jordan.manager@company.com",
  "first_name": "Jordan",
  "last_name": "Manager",
  "role": "BUSINESS_USER",
  "status": "INVITE_PENDING",
  "is_manager": true,
  "department_id": "29693a31-0f8d-4c91-be40-016bc0a6161c",
  "location_id": "1552c5a2-8889-4c91-b557-25419e0c616e",
  "manager_id": "517347b0-0841-4e3f-9b78-45cdb9642f5c",
  "scheduled_deactivation_date": null,
  "scheduled_invitation_date": null,
  "entity_id": "f3175311-29f5-41b3-800c-d05fd1d8a494"
}
```

The user starts in `INVITE_PENDING`. They transition to `USER_ACTIVE` once they click the invite link and complete onboarding — you'll receive the `users.invite_accepted` webhook at that point (see the [Webhooks guide](https://docs.ramp.com/developer-api/v1/webhooks)).

<Info>
**Guest users get an automatic 6-month expiration**

When `role` is `GUEST_USER`, Ramp auto-populates `scheduled_deactivation_date` 6 months out from the invite date. To opt out, pass `"scheduled_deactivation_date": null` explicitly on create.
</Info>

### Keep their profile in sync

Role changes, reorgs, moves between locations or entities, and manager changes all go through `PATCH /developer/v1/users/{user_id}`. Only the fields you include are updated; everything else is left untouched. A representative body:

```json
{
  "department_id": "4eab1943-1234-5678-9abc-def012345678",
  "location_id": "4350a152-a5e7-4617-bae8-16839beb00d0",
  "direct_manager_id": "78f823a0-44db-434e-a0e6-4bdb43950434",
  "is_manager": true,
  "role": "BUSINESS_BOOKKEEPER"
}
```

A successful PATCH returns `200 OK` with an empty body — re-fetch the user to see the applied changes.

Patchable fields: `first_name`, `last_name`, `role`, `department_id`, `location_id`, `direct_manager_id`, `is_manager`, `scheduled_deactivation_date`, `auto_promote`. Any other field — including `email`, `scheduled_invitation_date`, and `status` — is silently ignored.

To clear a nullable field like `direct_manager_id` or `scheduled_deactivation_date`, send `null` explicitly.

### Schedule their last day

If you know someone's end date ahead of time — a fixed-term contractor, a resignation with notice — PATCH `scheduled_deactivation_date` on the user. Ramp flips them to `USER_INACTIVE` automatically on that date; until then they keep working normally.

`PATCH /developer/v1/users/{user_id}` with `{"scheduled_deactivation_date": "2027-01-15"}`. A subsequent GET shows the date populated:

```json
{
  "status": "USER_ACTIVE",
  "scheduled_deactivation_date": "2027-01-15T17:00:00+00:00"
}
```

To reschedule, PATCH again with a new date. To cancel, PATCH with `{"scheduled_deactivation_date": null}`.

<Info>
**Date vs. datetime**

You submit `scheduled_deactivation_date` as a plain date (`YYYY-MM-DD`), but Ramp returns it as a full ISO-8601 datetime with a time-of-day derived from the business's local timezone. Compare against the date portion only when reading it back.
</Info>

### Deactivate them immediately

To deactivate a user synchronously, `PATCH /developer/v1/users/{user_id}/deactivate` with no body. The response is `200 OK`, empty body. The user is now `USER_INACTIVE`:

```json
{
  "id": "019db12c-faa3-772e-b874-7e8bce265bbb",
  "status": "USER_INACTIVE"
}
```

Once a user is `USER_INACTIVE`:

- All of their cards are frozen and new transactions decline.
- They can't log in to Ramp or receive notifications.
- Their historical transactions, receipts, and audit trail remain intact for admins.

Deactivation is idempotent — calling `/deactivate` on an already-inactive user is a no-op and still returns `200 OK`.

## Scenario 2: Deactivate and reactivate a user

When deactivation isn't permanent — a leave of absence, a contractor pausing between engagements, a mistaken offboarding — Ramp preserves the user record so you can bring them back with a single call. They keep the same `user_id`, cards, department, location, manager, role, and custom fields.

### Deactivate them now

`PATCH /developer/v1/users/{user_id}/deactivate` with no body returns `200 OK` and an empty body. The user becomes `USER_INACTIVE`; cards freeze and logins stop working. Historical data stays visible to admins.

### Reactivate them when they come back

`PATCH /developer/v1/users/{user_id}/reactivate` with no body returns `200 OK` and an empty body. A subsequent GET shows them back as `USER_ACTIVE`:

```json
{
  "id": "019db12c-faa3-772e-b874-7e8bce265bbb",
  "status": "USER_ACTIVE"
}
```

Previously issued cards are unfrozen and notifications resume. If their role or reporting line changed during the leave, follow up with `PATCH /users/{user_id}` to update those fields.

<Info>
**Reactivate instead of re-creating**

Do not call `POST /users/deferred` for a returning user. Reactivation preserves their spend history and configuration; creating a new record with the same email just returns the existing `user_id` anyway.
</Info>

## Scenario 3: Stage a draft user and schedule their invite

For new hires who start in the future — a class of interns starting next Monday, a contractor kicking off on the first of the month — you can stage the user in Ramp as a draft today and have Ramp send the invite automatically on a specific date and time.

Drafts:

- Exist as records in your business (you can assign department, location, manager up front).
- Do **not** receive an invite email.
- Cannot log in or spend.
- Appear in `GET /users?status=USER_DRAFT`.

### Create the draft

`POST /developer/v1/users/deferred` with `is_draft: true` in the body:

```json
{
  "email": "taylor.newhire@company.com",
  "first_name": "Taylor",
  "last_name": "Newhire",
  "role": "BUSINESS_USER",
  "department_id": "29693a31-0f8d-4c91-be40-016bc0a6161c",
  "location_id": "1552c5a2-8889-4c91-b557-25419e0c616e",
  "is_draft": true,
  "idempotency_key": "hris-sync-2026-04-21-taylor-newhire"
}
```

Poll the deferred task, then fetch the user — they start in `USER_DRAFT`:

```json
{
  "id": "019db12d-be50-718f-988c-d2ed5f7a93d0",
  "email": "taylor.newhire@company.com",
  "status": "USER_DRAFT",
  "role": "BUSINESS_USER",
  "scheduled_invitation_date": null,
  "scheduled_deactivation_date": null
}
```

### Schedule their invite

`POST /developer/v1/users/{user_id}/invite` with `action: SCHEDULE` and the time you want the invite to go out. `invitation_time` is an ISO-8601 datetime — include a timezone offset so there's no ambiguity.

```json
{
  "action": "SCHEDULE",
  "invitation_time": "2026-06-15T14:00:00+00:00"
}
```

Response: `200 OK`, empty body. The user stays in `USER_DRAFT` with the scheduled time populated:

```json
{
  "id": "019db12d-be50-718f-988c-d2ed5f7a93d0",
  "status": "USER_DRAFT",
  "scheduled_invitation_date": "2026-06-15T14:00:00+00:00"
}
```

When the scheduled time arrives, Ramp sends the invite email automatically and moves the user to `INVITE_PENDING`. Once they accept it, they become `USER_ACTIVE` and you can PATCH, deactivate, or reactivate them as normal.

### Reschedule, cancel, or send immediately

Plans change. `POST /developer/v1/users/{user_id}/invite` takes three actions:

| Action | Body | Effect |
| --- | --- | --- |
| `SCHEDULE` | `{"action": "SCHEDULE", "invitation_time": "<ISO-8601 datetime>"}` | Sets or updates `scheduled_invitation_date`. Calling it again overwrites the previous scheduled time. User stays `USER_DRAFT`. |
| `DESCHEDULE` | `{"action": "DESCHEDULE"}` | Clears the scheduled invitation. User stays `USER_DRAFT`. Idempotent — no-op if nothing is pending. |
| `SEND_NOW` | `{"action": "SEND_NOW"}` | Sends the invite email immediately and transitions the user to `INVITE_PENDING`. |

<Info>
**Invite actions only apply to draft users**

`SCHEDULE`, `DESCHEDULE`, and `SEND_NOW` are only valid when the target user is `USER_DRAFT`. Calling them on a user in any other state returns `400 DEVELOPER_7154`.
</Info>
